What it does

I built LuckTAB because most tab list plugins out there felt either bloated or too limited. Here's what's actually in it:

📋
Rank Prefixes & Sorting
Runs on LuckPerms, with anti-flicker updates so the tab list doesn't jump around.
🌈
Animated Gradients
Smooth RGB gradients via the <gradient:> tag, works everywhere in the plugin.
🎨
Hex Colors
Standard &#RRGGBB codes, in every text field.
📊
Scoreboard Sidebar
Configurable per world, with anti-flicker and smart change detection.
💬
Chat Format
Custom format per rank, full color support.
🌍
Per-World Config
Different designs depending on the world, with a fallback to default.
👻
Vanish Support
Works with Essentials, CMI, SuperVanish, and PremiumVanish.
📡
PlaceholderAPI
Any PAPI placeholder can be dropped in.
🔄
LuckPerms Migration
Sync prefixes between config and LuckPerms, weights calculated automatically.
📶
Ping Colors
%lucktab_ping_colored% — automatically green/yellow/red based on ping.

Installation

  1. Download the latest version from Modrinth
  2. Drop the .jar into your plugins/ folder
  3. Make sure LuckPerms is installed — nothing works without it
  4. Start or restart your server
  5. LuckTAB will generate plugins/LuckTAB/config.yml automatically
  6. Edit the config and apply changes with /lucktab reload

Quick Setup

Open plugins/LuckTAB/config.yml and set up your ranks:

prefixes:
  - "owner;<gradient:#FF0000:#FF7700>[Owner] </gradient>;;&#FF7700"
  - "admin;&#FF5555[Admin] &7;;&#FF5555"
  - "mod;ᖳFF[Mod] &7;;ᖳFF"
  - "vip;&6[VIP] &7;;&6"
  - "default;&7[Player] &7;;&7"

Format: luckperms_group;prefix;suffix;namecolor

PartWhat it does
groupExact LuckPerms group name (case-sensitive)
prefixShown in the tab list, nametag, and chat
suffixJust leave it empty with ;; if you don't need one
namecolorColor applied to the player's name itself

Setting Up LuckPerms

/lp user <player> group set owner
The group name has to match exactly what's in your prefixes list, or it just won't pick it up.

Migrating from LuckPerms

Already have prefixes set up in LuckPerms? You can pull them straight into your config:

/lucktab migrate from-luckperms

Or the other way around, push your config prefixes into LuckPerms:

/lucktab migrate to-luckperms

Prefix Source

ValueWhat it means
lucktabPrefixes come from the config (default)
luckpermsPrefixes are read directly from LuckPerms meta — respects inheritance

Per-World Designs

Add a section using the exact world name under designs. If nothing matches, LuckTAB falls back to default automatically.

World names are case-sensitive. Use /mv list or check your server folder to get the exact name.

Rank Priority

The order of entries in prefixes decides priority — top = highest. If a player has multiple groups, the topmost match wins.

❓ FAQ

My rank prefix isn't showing up

  1. Check that the group name in prefixes matches the LuckPerms group name exactly
  2. Check the player's group: /lp user <player> info
  3. Run /lucktab reload
  4. Top entry = highest priority — make sure owner is above default

The scoreboard is flickering

Make sure you're running Paper or Purpur. On Spigot, scoreboard lines using hex/gradient colors can flicker a bit — that's a Minecraft client limitation, not the plugin.

Gradients look choppy or jump

Usually happens with very short text — fewer characters means fewer color steps. Longer text gets a smoother gradient automatically, since the interpolation is sine-based.

%lucktab_rank% shows the raw gradient tag

That means LuckTAB can't find a matching rank for the player. Check the following:

  • Does the player actually have a LuckPerms group assigned?
  • Is that group in the prefixes list?
  • Does the spelling match exactly?

Does LuckTAB work on Folia?

No. LuckTAB uses Bukkit's scheduler, which isn't compatible with Folia. There are no plans to support it right now.

Does LuckTAB work on BungeeCord/Velocity?

LuckTAB is a per-server plugin. For network-wide player counts, use a PlaceholderAPI expansion that supports BungeeCord.

How do I report a bug?

Fastest way is via Discord: dc.gg/paperstream

What it's about

In short: the plugin constantly checks how loaded your server is (MSPT and how many chunks are loaded), and based on that, adjusts view and simulation distance per world automatically. No more manually tweaking things when everyone suddenly logs on in the evening.

Features

FeatureDescription
Dynamic Sim & View DistanceAutomatically adjusts per world based on server load
3 Adjustment ModesProactive, reactive, or mixed — your choice
MSPT-based PredictionA linear regression model prevents bad increases before they happen
Per-world ConfigurationEach world gets its own min/max, weight, and exclusion settings
Temporary OverridesLock distances for a set duration via command
PlaceholderAPI Support20+ placeholders for scoreboards, tab lists, and more
In-game AnnouncementsNotify staff on distance changes, format fully configurable
/vdt Command SuiteStatus, reload, enable/disable, manual override — all in one command

⚙️ Configuration

# ============================================================
#   ViewDistanceTweaks — LucasTHCR Edition
# ============================================================

version: 1

# Set to true once you've configured the plugin.
enabled: false

# Adjustment mode:
#   proactive — adjusts based on chunk count targets
#   reactive  — adjusts based on MSPT thresholds
#   mixed     — combines both; prioritizes reducing distance under load (recommended)
adjustment-mode: mixed

# Ticks to wait after startup before the first check runs.
# 2400 = 2 minutes — lets the server stabilize after a restart.
start-delay: 2400

# How often to run a check (in ticks). 600 = every 30 seconds.
ticks-per-check: 600

# How many consecutive checks must confirm an increase before it's applied.
# Higher = more conservative. Recommended: 8–15.
passed-checks-for-increase: 10

# How many consecutive checks must confirm a decrease before it's applied.
# 1 = immediate reaction to overload (recommended).
passed-checks-for-decrease: 1

# Log a console message whenever distances change.
log-changes: true

# Send an in-game message to players with the viewdistancetweaks.notify permission.
announce-changes-in-chat: false

# Format of the announce message. Supports & color codes.
# Placeholders: {world}, {oldSim}, {newSim}, {oldView}, {newView}
announce-format: "&7[VDT] &f{world}: Sim &e{oldSim}&7→&a{newSim} &f| View &e{oldView}&7→&a{newView}"

# ============================================================
#   PROACTIVE MODE
# ============================================================
proactive-mode-settings:

  # Maximum total ticking chunks across all worlds.
  # Formula: (2*sim+1)^2 * players
  # Example: 20 players at sim 8 → (17)^2 * 20 = 5780
  global-ticking-chunk-count-target: 5780

  # Maximum total non-ticking chunks (view distance beyond sim distance).
  # Formula: ((2*view+1)^2 - (2*sim+1)^2) * players
  # Example: 20 players, view 12, sim 8 → (625 - 289) * 20 = 6720
  global-non-ticking-chunk-count-target: 6720

# ============================================================
#   REACTIVE MODE
# ============================================================
reactive-mode-settings:

  # Below this MSPT, consider increasing distances.
  increase-mspt-threshold: 40.0

  # Above this MSPT, consider decreasing distances.
  # A full tick takes 50ms. Above ~47ms players start noticing lag.
  decrease-mspt-threshold: 47.0

  mspt-tracker-settings:

    # How many ticks of data to average for the MSPT calculation.
    # 1200 = 1 minute. Higher = more stable but slower to react.
    collection-period: 1200

    mspt-prediction:
      # Use historical MSPT/chunk data to block increases that would cause lag.
      enabled: true

      # How many minutes of history to keep for the prediction model.
      history-length: 30

      # Minimum data points before the prediction model kicks in.
      # Prevents bad decisions right after a server restart.
      min-samples: 5

# ============================================================
#   WORLD SETTINGS
# ============================================================
# Worlds not listed here inherit the 'default' settings.
# Use the exact world folder name as the key.

world-settings:

  default:
    simulation-distance:
      exclude: false
      min-simulation-distance: 4
      max-simulation-distance: 12

    view-distance:
      exclude: false
      min-view-distance: 6
      max-view-distance: 16

    # How much weight this world's chunks carry in global calculations.
    # 0.5 = half weight (e.g. The End, which has a lot of empty chunks).
    # 0.0 = excluded from the global chunk budget entirely.
    chunk-weight: 1.0

    chunk-counter-settings:
      # Count overlapping chunks from nearby players only once.
      exclude-overlap: true

    # Never reduce distances below the number of players currently in this world.
    never-reduce-below-players: false

    # Gives this world 50% more chunk budget in proactive calculations.
    # Useful for the main overworld on multi-world servers.
    priority-world: false

  # Example — Nether
  # world_nether:
  #   simulation-distance:
  #     min-simulation-distance: 4
  #     max-simulation-distance: 8
  #   view-distance:
  #     min-view-distance: 4
  #     max-view-distance: 10
  #   chunk-weight: 0.8

  # Example — The End (lots of empty chunks → lower weight)
  # world_the_end:
  #   simulation-distance:
  #     min-simulation-distance: 4
  #     max-simulation-distance: 10
  #   view-distance:
  #     min-view-distance: 6
  #     max-view-distance: 14
  #   chunk-weight: 0.5

  # Example — Main world with priority
  # world:
  #   simulation-distance:
  #     min-simulation-distance: 6
  #     max-simulation-distance: 12
  #   view-distance:
  #     min-view-distance: 8
  #     max-view-distance: 16
  #   chunk-weight: 1.0
  #   priority-world: true
Best to leave the defaults running for a bit and watch /vdt status before touching the thresholds.

Temporary Overrides

Using --duration you can lock a distance for a set number of minutes without touching the config — handy for events or quick testing:

/vdt viewdistance 10 world --duration 30

Once the duration is up, the plugin goes back to automatic control.

🔧 How It Works

The plugin measures MSPT (milliseconds per tick) every tick, and also looks at how many chunks players currently have loaded. Based on both, it decides whether simulation and view distance should go up, stay the same, or come down.

Proactive Mode

Targets a configured maximum chunk count and calculates the ideal distance directly — thinking ahead rather than reacting.

Reactive Mode

Simply reacts to MSPT crossing configurable thresholds. Simpler, more direct approach.

Mixed Mode (recommended)

Combines both approaches, and always prioritizes reducing distance when the server is under load — safety first.

The prediction model behind the scenes

A built-in linear regression model tracks historical MSPT/chunk-count pairs and blocks increases when it predicts the server would end up crossing the decrease threshold. That's what prevents the classic increase → lag → decrease loop.

It's worth giving the plugin a few minutes of observation time (start-delay) before it starts acting — right after a restart, the numbers are still settling.
